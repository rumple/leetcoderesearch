# LeetCode Autoresearch Loop

Karpathy's autoresearch methodology applied to LeetCode: one solution file, one empirical metric (Big-O complexity class), one baseline commit. Iterate until the theoretically optimal complexity is reached or the budget is exhausted.

## Core Invariants

- **One file**: the solution file leetgo generated (e.g. `solution.py`)
- **One metric**: empirically measured Big-O complexity class (ground truth), cross-checked by static analysis
- **One baseline**: the best solution committed to git so far
- **Loop budget**: max 10 iterations total, stop early if 5 consecutive failures or optimal complexity reached

---

## Phase 1: Workspace Setup

Determine the solutions directory:
```
SOLUTIONS_DIR = $LEETCODE_SOLUTIONS_DIR or ~/leetcode-solutions
PROBLEM_DIR   = $SOLUTIONS_DIR/<num>-<slug>/<language>/
```

The leetgo-generated solution file is already inside the leetgo workspace. Copy or work with it in place — do NOT move it, as `leetgo test last` and `leetgo submit last` operate on it.

For git tracking, work inside the leetgo workspace directory (wherever `leetgo pick` placed the file). Run `git init` there if it is not already a git repo.

## Phase 2: Naive Baseline

### 2a. Write the brute-force solution

Implement the simplest correct solution — no cleverness, no optimization. Nested loops, recursion without memoization, whatever is easiest to verify correct. The goal is a passing baseline, not a good one.

### 2b. Verify correctness

```bash
leetgo test last -L
```

If tests fail: fix the solution until they pass. This does NOT count as an autoresearch iteration — it is setup.

### 2c. Write gen_inputs.py

Write a problem-specific input generator based on the problem constraints. This file is written ONCE and reused across all iterations.

```python
# gen_inputs.py
import random

def generate(n: int):
    """Return valid input tuple for the solution function at scale n."""
    # Problem-specific — generated by Claude based on constraints
    # Example for two-sum:
    nums = random.sample(range(-10**4, 10**4), min(n, 20000))
    target = random.randint(-10**4, 10**4)
    return (nums, target)
```

### 2d. Write and run benchmark.py

```python
# benchmark.py — rewritten each iteration
import time, statistics, importlib.util, sys, pathlib

# Dynamically import the solution
spec = importlib.util.spec_from_file_location("solution", pathlib.Path(__file__).parent / "solution.py")
mod = importlib.util.module_from_spec(spec)
spec.loader.exec_module(mod)
solution = mod.Solution()

from gen_inputs import generate

SIZES = [100, 300, 1000, 3000, 10000]
RUNS  = 3

times = []
for n in SIZES:
    inp = generate(n)
    # Determine which method to call (first public method of Solution)
    method = next(m for m in dir(solution) if not m.startswith('_'))
    fn = getattr(solution, method)
    meds = []
    for _ in range(RUNS):
        t0 = time.perf_counter()
        fn(*inp)
        meds.append(time.perf_counter() - t0)
    times.append(statistics.median(meds))
    print(f"n={n:>6}: {times[-1]*1000:8.2f}ms")

print("\nGrowth ratios (t(3n)/t(n)):")
for i in range(len(SIZES)-1):
    ratio = times[i+1] / times[i] if times[i] > 0 else float('inf')
    print(f"  n={SIZES[i]}→{SIZES[i+1]}: {ratio:.2f}x")
```

Run it:
```bash
python3 benchmark.py
```

### 2e. Interpret the ratio table

| Ratio (t(3n)/t(n)) | Complexity class |
|--------------------|-----------------|
| ~1.0–1.2           | O(1)            |
| ~1.3–1.8           | O(log n)        |
| ~2.7–3.3           | O(n)            |
| ~4.3–5.5           | O(n log n)      |
| ~8.0–10.0          | O(n²)           |
| ~25–30             | O(n³)           |
| grows explosively  | O(2^n) or worse |

Take the **median ratio** across the size steps. Cross-check with static code analysis. If they conflict, trust the empirical result.

### 2f. Commit the baseline

```bash
git add .
git commit -m "baseline: <slug> O(n²) [empirical ratio 9.1]"
git tag <slug>-baseline
```

Record: `baseline_class = <inferred class>`, `failure_count = 0`, `iteration = 0`

## Phase 3: Autoresearch Loop

Repeat until termination:

### Step A: Check termination conditions

Stop if ANY of:
1. `baseline_class` == optimal for this problem type (consult `references/complexity_ladder.md`)
2. `failure_count` >= 5
3. `iteration` >= 10

### Step B: Form hypothesis

Read `references/optimization_hypotheses.md`. Select the most promising technique that:
- Targets a complexity class strictly better than `baseline_class`
- Is applicable to this problem's structure (access pattern, data shape, constraints)

State the hypothesis explicitly:
> "Hypothesis iter-N: Replace nested loops with a hash map to reduce O(n²) → O(n). Applicability: two-sum requires finding complement; hash map gives O(1) lookup."

### Step C: Implement

Edit the solution file with the new approach. Think through correctness before running tests — don't rely on tests to find logic bugs.

### Step D: Local test

```bash
leetgo test last -L
```

If FAIL:
- `git stash pop` to restore baseline
- `failure_count += 1`
- `iteration += 1`
- Go to Step A

### Step E: Benchmark

Rewrite `benchmark.py` if needed (the import path stays the same; the solution changed). Run:
```bash
python3 benchmark.py
```

Infer `new_class` from the ratio table.

### Step F: Submit

```bash
leetgo submit last
```

Parse the runtime percentile from output: `"faster than X% of <lang> online submissions"`.

### Step G: Evaluate

**If `new_class` strictly better than `baseline_class`:**
```bash
git stash drop 2>/dev/null || true
git add .
git commit -m "iter-N: <slug> <new_class> [ratio X.Xx] via <technique>"
git tag -f <slug>-baseline
```
Update: `baseline_class = new_class`, `failure_count = 0`

**Otherwise (same class or worse):**
```bash
git stash pop
failure_count += 1
```

Increment `iteration += 1`. Go to Step A.
